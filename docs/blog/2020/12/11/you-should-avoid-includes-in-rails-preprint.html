You should avoid `includes` in Rails [PREPRINT]2020-12-11
This post is not meant to be shared yet, it is only here to make it easier to gather feedback. Thank you

includes is the recommended way to load associations of your records eagerly. In fact, the Rails guide for eager loading only mentions includes.

But is there are 2 other methods: preload and eager_load which have the same goal of eager loading data, but with different means and possible side effects.

Here's why you should avoid includes:

  • includes makes it easy to introduce an odd bug
  • There is an alternative method that cannot introduce the bug

What bug? Lets show the introduction of an unexpected bug:

Say you have these records:

p = Post.create!
p.comments.create!(content: 'hello', spam: true)
p.comments.create!(content: 'world')

You want posts that have at least one comment marked as spam.

# distinct because joins can make duplicates
posts = Post.joins(:comments).distinct
            .where(comments: {spam: true})

How many comments in this array?

posts.first.comments.to_a.size

Answer

2

You will need to display the comments of the posts, so you includes them.

How many comments are in the array?

posts.includes(:comments).first.comments.to_a

Answer

Only 1!

What's the count?

posts.includes(:comments).first.comments.count

Answer

2

What about size?

posts.includes(:comments).first.comments.size

Answer

1

Personally, I find that unexpected.

Imagine if the post is passed to a view or a helper, which uses the comments. It would only print the comments that matched the condition. Now try to debug from that helper, you would see that post.comments only has 1 comment.

Hopefully you know that this is how includes (and eager_load, see below) behaves, so you may look up where the post comes from and figure it out. Good luck otherwise.

This is considered a feature. I consider it a maintenance burden. At least the documentation doesn't recommend using it.

The alternative

So what should you use when you want to avoid the dreaded N+1 query problem?

includes has 2 modes of operation, it either:

  • calls preload, which will do a 2nd query to load all of the associated records.

  • calls eager_load if your query refers to the table of an association passed to includes. This will change the query to do JOIN on the table.

    If you reference that table with a condition, you will only get records with the matching association and only eager load the associated records that match the condition. This is what causes the bug.

That switch to eager_load is what caused the bug above. I'd say includes couples the eager loading to the logic of filtering the data we want. preload doesn't do that.

preload only does eager loading, the same way all the time. Don't risk includes doing something wrong just use preload all the time; it works like includes, without the booby trap.

posts = Post.joins(:comments).distinct
            .where(comments: {spam: true})
posts.preload(:comments).first.comments.to_a.size #=> 2
posts.preload(:comments).first.comments.count #=> 2
posts.preload(:comments).first.comments.size #=> 2

But includes is everywhere...

So why is includes the popular way of eager loading data? I think it's for two reasons:

  • It's the "smart" function out of the 3, it will pick the "right" strategy when needed, and since it usually picks the safer preload, then most don't get hurt by it.
  • includes as been the recommended one from the start, so most use it and are familiar with it, so most recommend it.

When is eager_load needed?

Rarely...

Maybe some use it to reduce the number of queries when they do eager loading. I don't think it really saves much, and there is a risk of slowing things down by making queries that are heavier (lots of joins).

I mostly see it used to do a where on an association. You may see a recommendation to use includes, and then have a condition on that table. This is actually using eager_load and, as mentionned at the begining, this can cause subtle bugs.

It looks like this:

# Please stop doing this :(
Post.includes(:comments).where(comments: {spam: true})
Post.eager_load(:comments).where(comments: {spam: true})

Problems with includes/eager_load for this:

  • The partial loading of has_many associations, as mentionned above is unexpected / confusing
  • If you don't need the association to be loaded it is wasteful to do so
  • Doesn't compose well
  • Doesn't handle recursive associations (ex: parent/children)

Truth is, this need for a where on an association is not something that ActiveRecord handles well. Avoiding the includes / eager_load, the only remaining options is to use joins. But there are still multiple problems with it.

It looks like this:

# Please stop doing this :(
Post.joins(:comments).where(comments: {spam: true})
Post.joins(:comments).distinct.where(comments: {spam: true})

Using joins like this is better than includes / eager_load, but still far from great:

  • Doesn't handle recursive associations (ex: parent/children)
  • Doesn't compose well with other similar conditions
  • Doesn't interact work well with or

What I recommend is to instead use a gem I made just for this purpose: activerecord_where_assoc.

# Please consider doing this:
Post.where_assoc_exists(:comments, spam: true)
# Or using a scope such as is_spam:
Post.where_assoc_exists(:comments) { is_spam }

How much better?

  • It just adds a single where condition, so it composes well and works with or
  • Handles recursive associations (ex: parent/children)
  • Can handle polymorphic belongs_to (includes and joins would simply refuse)
  • Easy to do a NOT of the condition (I.E: Where no comment has the content 'hello')
  • Composes with other such queries, even on the same association, even with negations
  • Works well with the or method

One more reason against includes

If you are just going to end up doing a preload behind the scene, then using includes is slower since there are extra checks before running the request. Testing with a trivial case, I get a consistent 1 to 2% improvement when using preload. Minor? Sure. But it's free!

Here is the micro benchmark:

Benchmark.measure { 50000.times { Post.includes(:comments).to_a } }
Benchmark.measure { 50000.times { Post.preload(:comments).to_a } }
